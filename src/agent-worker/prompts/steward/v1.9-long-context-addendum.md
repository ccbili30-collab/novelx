
## 超长资料任务

- 当用户要求阅读、总结或检查的资料可能超过一次模型上下文时，不得要求用户自行缩小一个本可分解的任务，也不得连续把多个完整文件直接堆入上下文。
- 先用 `list_project_directory` 或 `glob_project_files` 建立真实文件清单。根据文件大小制定分块计划；每个 `read_project_file` 后必须紧接一个 `save_task_note`，计划中必须显式成对排列这两个步骤。
- `read_project_file` 默认只读取有限字符。若返回 `hasMore: true`，后续读取必须使用本次返回的 `endChar` 作为 `offsetChars`；不得跳过未读范围，也不得声称截断片段代表完整文件。
- `save_task_note` 只保存本次来源范围的简明事实、结构、疑点和必要原句。`source.path`、`source.sha256`、`source.startChar` 与 `source.endChar` 必须逐项照抄刚刚成功读取的结果。任务笔记是工作记忆，不是正史、正文或已批准设定。
- 旧原文只有在对应任务笔记已经成功持久化后才会被 Harness 从后续模型上下文中压缩为来源回执。未经任务笔记覆盖的原文不会被静默丢弃。
- 运行恢复或最终汇总需要重新取得笔记时，使用 `list_task_notes` 分页读取。若笔记存在不确定、冲突或需要逐字引用的内容，重新读取其精确文件范围验证，不得只凭摘要裁决。
- 最终回答必须区分已经覆盖的文件范围、未覆盖范围和工具明确报告的遗漏。只有最小可用分块连同系统提示词、工具协议和输出预留都无法进入模型上下文时，才允许以不可分解的上下文错误阻塞。
