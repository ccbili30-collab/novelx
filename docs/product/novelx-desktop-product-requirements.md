# NovelX Desktop 完整产品需求

状态：长期产品 Source of Truth（事实来源）

范围：NovelX 桌面端；Android 仅作为未来消费与同步端讨论

最后校准：2026-07-14
注意：本文件描述最终产品和演进边界，不代表当前代码已完成这些能力。真实完成度见 `docs/project/current-state-and-routes.md`。

## 1. 产品定义

NovelX Desktop 是一个面向小说、世界观、OC（原创角色）、互动故事和可视书的 Agent-native（智能体原生）创作工作台。它把通用编码 Agent 的工作方式映射到创作领域：能读项目、查资料、制定计划、调用工具、安排专业 Agent、形成变更、运行检查、生成产物、回溯版本，并把长期讨论转化为可复用的世界包和故事项目。

用户不需要先理解文件格式、数据库、图数据库、向量检索、Prompt（提示词）或版本控制。用户主要面对 Steward（大管家）：

- “和我一起把这个世界为什么有精灵聊清楚。”
- “把这个 OC 放进另一个世界，但保留她的底色。”
- “按搞笑但不轻浮的风格写第一章，并检查有没有把前面写成悲剧。”
- “把这本现成小说拆成世界包、角色卡和多个可游玩的起点。”
- “给刚才的角色做立绘，给下一回合生成一张场景图。”
- “这里和三个月前的设定冲突，列出来源并让我选择。”

大管家负责理解目标、检查项目、检索证据、建立 Goal（目标）和 Plan（计划）、安排 Writer、Checker、GM、Decomposer、Image Agent 等专业能力，并将结果以正文、资源、关系、图片、图谱、Change Set（变更集）或玩家回合呈现。NovelX 是大管家的 Harness（智能体运行框架），不是大管家之外再套一层装饰性聊天 UI。

## 2. 核心价值

### 2.1 让创意在聊天中生长而不失控

用户的想法会随着长期聊天发生巨大变化。系统不能把“最初设定”当成永恒真理，也不能让后来的随口一句话静默覆盖整个世界。每次沉淀都应区分：讨论、草稿、候选事实、已接受设定、故事范围正史和玩家时间线；出现冲突时，大管家主动说明“哪里不同、各自来源、影响哪些内容”，再按 Free / Assist 规则处理。

### 2.2 让模型真的记得，而不是假装记得

长期记忆不是无限拼接聊天记录。NovelX 保存原始来源、结构化对象、权威断言、事件关系、任务笔记、版本和检索凭证。上下文不足时，Harness 主动检索文档和图谱；压缩时保留 Goal、Plan、工具配对、未完成审批、来源版本和冲突；回答涉及既有事实时优先引用来源，不足则明确说不知道或继续查找。

### 2.3 让世界、角色和故事成为可组合资产

创作者可以拥有多个 World Base（世界基底）、OC Base（角色基底）和 Story Project（故事项目）。故事可以版本化引用一个或多个世界与多个 OC；同一 OC 在不同故事中可以形成 OC Variant（角色变体），保留底色但拥有不同经历、能力和关系。世界与角色可交叉复用，不被某一本小说永久占有。

### 2.4 让作品从文本变成可视、可玩、可回溯的世界

正文、角色立绘、场景图、地图、物品、力量体系、事件图谱和玩家状态要形成可交互的作品视图。用户可以只写小说，也可以边聊边创造世界，或进入第一人称玩家模式。所有模式使用同一项目知识和版本链，但根据视角限制可见信息。

## 3. 目标用户与主要任务

| 用户 | 主要需求 |
| --- | --- |
| 世界观创作者 | 从天文地理到人文历史持续讨论、整理、关联和检索，不被固定人物卡模板限制。 |
| OC 创作者 | 创建角色底色、外观、能力、关系和延生故事，并让角色在不同世界形成变体。 |
| 小说作者 | 讨论大纲、风格和约束，编写正文，反复修改，检查连续性和来源。 |
| 互动故事玩家 | 用第一人称行动推动世界，由 GM 裁决、Writer 表现、Checker 约束，并可看到图片和公开状态。 |
| 世界包制作者 | 导入作品或建立可复用世界包、角色卡和起始模板，供其他故事或移动端游玩。 |

同一用户可以在这些身份之间自由切换，项目不应强迫其先选择“作者”或“玩家”后永久锁定。

## 4. 核心对象模型

### 4.1 顶层可复用对象

- **World Base**：宇宙规则、地理、历史、文化、文明、种族、力量体系、地点、势力、物品、语言、生态、宗教、社会与其他设定的容器。
- **OC Base**：角色不可轻易丢失的底色，例如核心人格、视觉身份、原始动机、代表能力和创作者意图。
- **Story Project**：一条具体故事或创作工程，引用世界和 OC，拥有自己的主题、风格、结构、正文、时间线和故事范围正史。
- **Source Library（来源库）**：导入小说、散乱材料、图片、访谈笔记和外部世界包的原始证据层。
- **World Package（世界包）**：可导出、可版本化的世界/OC/规则/起始模板集合，不必携带原著完整剧情。

### 4.2 故事内部对象

- 卷、章、节、场景、段落与正文文档。
- 故事大纲、主题、叙事视角、风格约束、禁区和 Writer 指南。
- Story-scoped Canon（故事范围正史）、事件时间线、角色关系和未解决伏笔。
- Story Profile（故事配置）：一次可游玩故事所使用的故事版本、世界、OC 变体和规则集合。
- Start Profile（起始模板）：某个世界和故事配置的可游玩开局，不等同于原著既定未来。
- Playthrough（玩家流程）：玩家行动、GM 裁决、正文、公开状态与所固定的正史版本。

### 4.3 世界与 OC 的多层内容

世界和 OC 不能被限制为几张表单。两者都支持任意层级知识文档、结构化字段、关系、时间线、图像和小故事。结构化字段用于检索、检查和 UI；自由文档保存复杂表达。系统允许大管家在聊天后提出新分类，不要求所有世界都套用同一模板。

### 4.4 版本化关系

至少支持：`uses_world`、`uses_oc`、`has_variant`、`located_in`、`member_of`、`related_to`、`causes`、`contradicts`、`derived_from`、`depicts`、`appears_in`。关系必须带来源、作用范围和版本；不能只保存无出处的边。

## 5. 三种主要工作方式

### 5.1 Agent Mode（智能体模式）

这是以交流和任务为中心的主界面，形态接近 Codex：

- 左侧：项目目录与会话；每个会话是独立 Agent，上下文和私有任务记忆隔离。
- 中间：用户与大管家对话、Goal、Plan、审批、冲突、引用和最终结果。
- 右侧：默认打开“文件夹内容”；“活动与产物”独立折叠。大管家正在读什么、改什么、生成什么都能展开审计。
- 底部：Free / Assist、模型选择、上下文预算与输入。

Agent Mode 可以完成世界观讨论、OC 创作、小说写作、项目整理、导入拆解、图片生成和版本操作。项目底层文件可以由受权限控制的工具访问，但默认 UI 展示领域对象和小说内容，不把用户丢进代码编辑器。

项目管理必须提供：添加/打开目录、从列表移除、恢复、重新扫描和清理测试项目。会话管理必须提供：新建、重命名、归档、恢复、清空/删除历史、导出，以及只对上一条尚可安全重放的用户输入进行编辑重发。打开多少项目，左栏和右侧“文件夹内容”就必须反映多少真实项目，不能显示固定模板目录。

### 5.2 IDE Mode（创作编辑台模式）

- 左侧：世界、OC、故事、章节、地点、势力、图像、来源等多层资源树。
- 中间：正文编辑器、世界文档、角色卡、图谱、时间线、地图或图片视图。
- 右侧：与当前对象绑定的大管家对话、活动和产物。

用户手写与 Agent 修改进入同一版本链。UI 可以模板化创建常见对象，但模板不是数据模型的上限。

### 5.3 Player / Card Mode（玩家/卡片模式）

玩家可以第一人称行动，也可以把卡片模式当作“只看正式正文”的阅读/创作视图。每一回合由真实 GM 读取世界、当前状态、记忆、幸运值和玩家行动进行裁决；Writer 只润色不可变裁决；Validator / Checker 检查越权、来源和泄密。

玩家界面只显示正式正文、玩家动作、公开状态、选择和来源绑定场景图。Creator Lens 的隐藏事实、GM JSON、内部资产 ID、工具日志和未来剧情不得泄露。

## 6. 新项目、现有项目与开场

### 6.1 空目录

初始化标准领域目录和项目数据库，创建大管家会话。开场问题只建立最小世界底子；最后提供两条路：直接进入第一幕，或“和大管家一起创造”，继续深入讨论世界、OC、历史与规则。

### 6.2 散乱目录

先只读扫描 TXT、Markdown、DOCX、EPUB、图片和可识别的项目文件，建立 Source Library。大管家给出分类和导入计划；在 Assist 模式下经确认后形成资源与关系。扫描不能擅自改名、移动或删除原材料。

### 6.3 现成世界包或小说

用户可以打开已完成世界包，也可以导入小说并由 Decomposer（拆解器）分阶段提取人物、世界规则、事件、风格和候选起点。原著发生过的故事与“世界本来允许什么”必须分开；玩家时间线不能被原著未来强制覆盖。

## 7. 大管家与专业 Agent

### 7.1 大管家的职责

- 理解用户意图并选择直接回答、执行工具、创建 Goal / Plan 或委派任务。
- 检索项目与来源，不能因为 Prompt 里没有资料就瞎编。
- 把对话提炼为候选结构，解释冲突和影响。
- 管理项目写作风格、约束与 Agent 指南，例如给 Writer 建立可版本化的写作规范，而不是只靠当前聊天提醒。
- 安排专业 Agent 并校验其 Artifact（产物）。
- 在 Free / Assist、权限、预算和版本规则内推动变更。
- 用自然语言报告结果，同时提供可展开的活动、引用和变更证据。

### 7.2 专业 Agent

- **Writer**：根据故事目标和不可变事实写正文，不改写 GM 裁决。
- **Checker / Validator**：提取候选事实、检测冲突、检查连续性、风格、来源和玩家可见边界；不自行补剧情。
- **GM**：在玩家模式裁决行动与世界反应。
- **Decomposer**：拆解长篇来源，分块提取并保留定位。
- **Image Agent**：根据视觉身份锚点、场景事实和预算生成图片任务。
- **Research / Librarian**：读取、搜索、归档和构建来源索引。

多 Agent 不是数量展示。只有任务独立、文件/资源不重叠且并行收益明显时才并发。子 Agent 接受有界 Assignment（分配任务），默认只读或只提交 Change Set；不能扩大权限、完成父 Goal 或直接写正史。

## 8. Free / Assist 与正式接受

### Free（自由）

大管家可以在策略允许的低风险范围内自动接受、整理和提交，但所有动作仍可审计、可回溯。重大删除、跨项目操作、来源冲突和高风险正史替换不能因 Free 而无限授权。

### Assist（协助）

大管家先提出结构化变更，弹窗说明“写入哪里、作为草稿还是正式内容、影响什么、与什么冲突”，由用户确认、修改或拒绝。用户也可以只接受部分变更。

### 内容状态

至少区分：对话想法、工作草稿、候选事实、已接受知识、Canonical Assertion、故事范围正史、玩家流程事实。不能用一个“已保存”掩盖这些语义差异。

## 9. 长期记忆、文档与图谱

### 9.1 存储分层

```text
原始来源 / 完整文档
  -> 候选事实与实体关系
  -> 来源定位与版本
  -> 冲突检测
  -> Checker / 用户决策
  -> Canonical Assertion
  -> 图谱、时间线与检索索引投影
```

必须分开保存：原始来源、候选事实、已确认正史、故事范围正史、OC 变体、Agent 私有任务记忆、项目共享记忆和玩家已知事实。向量相似度、摘要或模型置信度不能自动把候选事实升级为正史。

### 9.2 图谱的产品表现

用户可以打开类似神经元网络的世界图谱，看到人物、地点、事件、规则、势力和因果关系；点击节点查看摘要、原文来源、时间、相关对象和版本。图谱支持筛选故事、时间、对象类型和 Creator / Player Lens。

图谱既用于展示，也用于 Agent 检索，但不是唯一真相数据库。完整叙述留在文档；断言用于一致性；图谱用于关系、导航和召回。

### 9.3 主动检索与压缩

Harness 在以下情况主动检索：用户问到已有设定、准备修改已存在对象、生成正文依赖旧事实、Checker 发现可能冲突、上下文压缩后恢复任务、玩家回合需要世界与最近记忆。

上下文压缩必须保留：Goal、Plan、Assignment、ToolCall / ToolResult 配对、未完成审批、Change Set、来源版本、检索凭证、Canonical Assertion、世界规则、冲突与恢复状态。长文档按可恢复游标分块，由 Runtime 调度，不由模型自己维护偏移量。

## 10. 版本、分支与“小说 Git”

NovelX 借鉴 Git 的思想，而不是把创作者暴露给 Git 命令：

- **Commit / Checkpoint**：一次可解释的正文、设定、关系和资产变更。
- **Diff**：按段落、字段、关系、事件和图片来源展示变化。
- **Branch**：探索不同设定、故事路线或玩家结果。
- **Merge / Reconcile**：检测语义冲突，由大管家和用户决定，不静默覆盖。
- **Blame / Provenance**：知道一句设定、一个事件或一张图来自哪里、哪个版本和哪个会话。
- **Tag / Release**：标记世界包、故事版本和可游玩发布版本。

回溯默认创建新的可审计头或分支，保留后来历史。会话分支、项目版本分支和玩家流程分支是三个不同概念，不能用一次“撤回”同时隐式改变三者。存储应使用事件、差异和内容寻址资产，避免每个节点复制整个项目。

## 11. 图像、地图与可视书

### 11.1 图像能力

- 文本 Provider 与图片 Provider 独立配置、测试和审计。
- 支持角色头像、半身/全身立绘、场景、物品、地图、背景与封面。
- Image Job（图片任务）异步排队，状态为 queued、generating、ready、failed、stale 或 reconciliation_required。
- 每张图片绑定来源资源、来源版本、用途、视觉身份锚点和生成配置；来源改变后图片可标记过期。
- 用户可以查看、重试、替换、接受和控制生成预算。
- 玩家阅读时可以在后台准备下一场景图片；只有 ready 且与当前故事/场景来源严格匹配的图片才展示，不用占位图冒充。

`codex-imagen2` Skill 和测试代理可用于开发验收，但正式产品必须通过自身 Provider 配置、安全存储、任务队列和资产仓库运行，不能依赖 Codex 会话 Skill 作为生产链。

### 11.2 可视化效果

作品预览应能联合展示主视觉、正式正文、角色卡和事件图谱。地图、物品和力量体系未来也应成为可点击资源。长期可以加入受权限约束的角色动画、技能特效和“翻到下一张卡时出现 OC”的演出，但脚本不能直接嵌入小说执行；这属于未来 Mod / Plugin（模组/插件）和视觉运行时。

## 12. 玩家模式与创作模式的协调

玩家流程固定开始时所使用的世界、故事、OC 变体和 Canon 版本。用户可以中途回到大管家长期讨论并修改世界：

- 不影响当前存档的修改可以继续使用。
- 影响未来但不改变已经发生事实的修改，可经说明后更新流程基线。
- 与已经发生事实冲突时，系统给出三类选择：继续旧存档正史、从新设定创建分支/新存档、或由用户批准一个明确的过渡/调和 Change Set。

系统不能自动补一段“中间剧情”来掩盖任何冲突。只有用户明确要求桥接故事时，Writer 才能在 GM/Canon 约束下创作。

## 13. 状态系统

状态是可选领域模块，不强迫所有小说游戏化。项目可以定义生命、体力、关系、声望、时间、天气、资源或自定义指标；数值变化必须来自 GM 裁决或正式事件，UI 可用图形、动画、颜色和卡片表现。纯文学项目可以关闭数值层，只使用叙事状态和事件。

## 14. 编辑、审查与展示

- Markdown / GFM、Mermaid 和结构化图表可渲染，但正式资源不以裸 Markdown 文件作为唯一产品视图。
- 引用显示文档名、原文片段、结构化位置和稳定版本；点击回到真实位置。
- ToolCall、Artifact、Change Set、冲突、审批和错误使用结构化组件，不靠正则猜文本。
- 用户消息支持复制；只有上一条尚未被后续用户消息覆盖的输入可撤回编辑并重发，避免隐式重写历史。
- 错误必须区分网络、限流、Provider、上下文、工具参数、权限、协议、版本冲突和程序错误，并显示失败步骤、自动重试和恢复入口。
- “已处理”可展开查看工具和产物，但不展示模型私密思维链。

## 15. 模型与上下文配置

- Provider 明确返回能力时采用真实上下文上限，不使用统一 12,800 假定。
- Max Tokens 默认 Auto，根据系统提示词、工具协议、会话历史、检索资料、输出预留和模型限制动态计算；不发送虚假的无限值。
- 设置显示上下文组成和剩余输入预算。
- 保存前提供连接测试与轻量 `ping`；`ping` 不自动加载完整项目上下文。
- 支持按 Steward、Writer、Checker、GM、图像和嵌入/重排选择角色模型；切换只影响下一 Run，并被审计。
- API Key 只在应用设置与操作系统安全存储中保存，不写源码、日志、项目或文档。

## 16. 导入、世界包与生态

### 导入

扫描 TXT、Markdown、DOCX、EPUB 和图片；保留原文件、哈希、结构位置与版权/来源信息。Decomposer 分阶段提取人物、规则、事件、风格和候选关系。长篇任务可中断恢复；每一步都有覆盖范围，不能把读取片段说成读完整本书。

### 世界包

世界包可包含 World Base、OC Base、可选故事、规则、图片和多个 Start Profile。导出前选择 Publish Profile（发布配置），决定哪些 Creator Lens 内容、版权来源和资产可公开。桌面端未来与 Android 同步，用手机进行卡片游玩和可视书阅读。

### 生态

市场、多人协作、群聊和插件生态均为后续能力。插件必须声明权限，不能直接访问密钥、任意执行项目脚本或写正史。现阶段不以这些功能阻塞核心创作闭环。

## 17. 视觉与交互方向

- Windows 桌面端优先，整体信息架构参考 Codex，但做领域等量替换，而不是复制代码编辑器。
- 默认白色清晰主题；保留 `cloude` 纸张主题、深色和高对比度主题。颜色、字体、间距、边框和阴影使用 NovelX Design Tokens（设计令牌）。
- Snow（猫）作为可选默认背景，可在设置中选择 Snow、自定义图片或无背景。宠物功能未来只订阅公开运行事件并播放表现，不拥有 Agent 权限。
- 当前猫形象同时作为 NovelX 的应用图标与 README 视觉标识；更换品牌资产时必须保留可回滚资源和安装包图标验收。
- 右侧滑动式历史、Goal / Plan、会话分支、批注、模型调整、活动与产物、项目内容都要来自真实持久状态。
- 多项目必须展示各自内容和会话；项目可移除、恢复、重扫，测试项目必须自动清理。
- 性能目标是编辑、切换、滚动和输入无明显卡顿；图谱和大型文档采用增量加载与虚拟化。

## 18. 非功能要求

### 可靠性

- 程序或 Worker 崩溃后不重复调用 Provider、重复收费、重复执行工具或重复提交 Change Set。
- ToolCall 与 ToolResult 严格配对；每次请求发送前本地校验协议。
- 外部副作用结果未知时进入 reconciliation_required，不自动重试。
- 写入使用版本前置条件、事务和单项目队列。

### 性能与成本

- 按需检索，不把整个项目注入每次请求。
- 文档分块、摘要、嵌入和图谱索引可后台执行。
- 图片、模型和 Agent 并发受预算控制。
- 不因节省 Token 使用假实现；优先减少重复上下文、重复测试和重复 Agent。

### 安全与隐私

- Electron 保持 contextIsolation、sandbox，关闭 nodeIntegration；Renderer 只使用 allowlisted typed IPC。
- 工具限制项目根、路径穿越、符号链接和副作用权限。
- 日志脱敏；图片与文档来源可审计；发布前检查版权与私密内容。

### 可维护性

- 公开协议版本化；数据库迁移可回滚或有恢复方案。
- 上游参考通过适配层隔离，保留许可证和来源说明。
- Prompt 有版本、发布门、回滚、契约测试和真实 Provider Eval。
- Windows 安装、应用内更新、崩溃恢复和诊断是正式发布能力。
- NovelX Desktop 保持独立仓库和独立发布线，不与 Android 源码混为一个工作树；两端通过明确的世界包/同步协议连接。

## 19. 端到端产品验收

一个“基础但真实”的 NovelX 至少要能完成：

1. 用户打开空项目或现有材料目录。
2. 与大管家讨论一个世界和一个 OC。
3. 大管家读取真实项目，提出结构化记录；Free 自动或 Assist 确认后进入版本链。
4. Checker 建立候选事实、来源和关系，冲突不会静默覆盖正史。
5. 大管家创建故事配置并让 Writer 生成正式正文。
6. Image Agent 使用独立真实 Provider 生成角色或场景图，资产绑定来源与版本。
7. 作品视图同屏展示正文、图片、角色和事件图谱。
8. 用户从作品进入玩家模式，GM 与 Writer 用固定世界/故事/OC 完成一回合。
9. 用户修改早期输入或设定时形成会话/项目/玩家分支的明确处理。
10. 关闭程序并重启后，项目、Run、Goal、Plan、来源、版本和未完成工作可恢复，不重复外部副作用。

其中任何模型内容、图片、工具活动或成功状态若来自 Fixture、Mock、模板或本地 reducer，都不能算 Live 验收。

## 20. 长期阶段路线

1. **桌面技术基础闭环**：Electron、React、TypeScript、SQLite、真实 Provider、安全配置、Change Set、图谱、版本、安装更新。
2. **多项目多会话工作台**：Agent / IDE 模式、项目注册、独立 Agent、共享与私有记忆、写入队列。
3. **完整创作对象与编辑**：世界、OC、故事、章节和任意知识层级；关系、约束与统一版本链。
4. **长期记忆、图谱和 Checker**：来源、候选事实、Canonical Assertion、冲突、Creator Lens、主动检索。
5. **世界到故事、分支与玩家模式**：World Base、Story Project、OC Variant、Story-scoped Canon、真实 GM / Writer / Checker。
6. **导入小说与世界包**：多格式来源、分阶段拆解、人工确认、多个起始模板、原著与玩家时间线分离。
7. **图像、地图和可视书**：异步图片、身份锚点、资产版本、预算、场景演出。
8. **发布、同步与生态**：世界包导出、Android 同步、版权、升级签名、崩溃恢复；之后再考虑市场、协作和插件。

阶段编号描述产品依赖，不等于当前代码完成度。Rust Runtime V2 是贯穿阶段的可靠性地基，不另算一个可被 UI 绕过的“后台阶段”。

## 21. 明确非目标与未决产品决策

当前不承诺：把 NovelX 变成通用代码 IDE；无限递归 Agent；任意脚本直接嵌入小说；静默跨会话共享私有聊天；无审计的自动正史；首版多人实时协作；首版插件市场；首版 iOS。

仍需未来用户决策或验证：商业模式、世界包版权策略、多人协作权限、插件沙箱强度、数值状态的默认模板、公开市场审核、Android 同步冲突策略、图像风格模型与成本上限。架构必须预留边界，但不能在未决定前假装已有产品答案。
